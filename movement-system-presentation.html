Camera Movement System

A declarative, data-driven architecture for camera input configuration

🎯 The Problem

Scattered boolean flags, hardcoded input logic, framerate-dependent inertia, and no clean way to customize input mappings.

✨ The Solution

A two-layer system: Input Mapping (what gesture means what) and Interaction Handlers (what that does to the camera), with framerate-independent movement.

📣 Community Driven

Directly addresses long-standing forum requests for input customization and delta-time bugs.

🔒 Backward Compatible

Opt-in via useMovementSystem. Legacy code path remains untouched.

New Architecture

Two composable layers, each answering one question

┌─────────────────────────────────────────────────────────┐ │ Physical Input (mouse, keyboard, touch, wheel) │ │ ↓ │ │ Layer 1: Input Mapping │ │ "What interaction does this gesture mean?" │ │ Declarative map: source + conditions → interaction │ │ ↓ │ │ Layer 2: Interaction Handlers │ │ "What does this interaction do to the camera?" │ │ Keyed by interaction name (source-agnostic) │ │ ↓ │ │ CameraMovement.computeCurrentFrameDeltas() │ │ Velocity model → framerate-independent deltas │ │ ↓ │ │ Camera state updated (alpha, beta, radius, target) │ │ ↓ │ │ Behavior<Camera> → post-input modifications (unchanged) │ └─────────────────────────────────────────────────────────┘

Type-safe input map entries

// Discriminated union — compiler prevents invalid combos
type InputMapEntry =
  | { source: "pointer";  interaction: T; button?: number;  modifiers?: InputModifiers; sensitivity?: number }
  | { source: "wheel";    interaction: T;                  modifiers?: InputModifiers; sensitivity?: number }
  | { source: "touch";    interaction: T; touchCount?: number; }
  | { source: "keyboard"; interaction: T; key?: number[]; modifiers?: InputModifiers; sensitivity?: number }

// ❌ Compile error — touchCount doesn't exist on pointer source
{ source: "pointer", touchCount: 2, interaction: "pan" }

Before vs. After

Swapping left-click from rotate to pan

❌ Legacy — Not cleanly possible
// Option A: Change panning mouse button
// But this only controls which button pans —
// left-click ALWAYS rotates in the pointer input
camera.attachControl(canvas, true, true, 0);
// panningMouseButton = 0 (left)
// But now what does right-click do? Nothing pans.
// And keyboard ctrl still pans, not rotates.

// Option B: Override checkInputs... 😰
// Subclass the input, rewrite the onTouch method,
// reverse the _isPanClick logic, hope nothing breaks
✅ New — One config change
// Just describe what you want:
camera.movement.inputMap = [
  { source: "pointer", button: 0,
    interaction: "pan" },      // Left = pan
  { source: "pointer", button: 2,
    interaction: "rotate" },   // Right = rotate
  { source: "wheel",
    interaction: "zoom" },
  { source: "keyboard",
    modifiers: { ctrl: true },
    interaction: "rotate" },   // Ctrl+keys = rotate
  { source: "keyboard",
    interaction: "pan" },       // Keys = pan
];

The forum thread (#19878) proposed an enum CtrlKeyBehaviours, a swapPanAndRotate boolean, or a useCtrlForRotation flag. With the new system, no new API surface is needed — it's just data.

Before vs. After

Keyboard: default arrows pan, ctrl+arrows rotate

❌ Legacy — Hardcoded if/else chain
// Inside ArcRotateCameraKeyboardMoveInput.checkInputs()
// Every key has the same branching pattern:
if (this._ctrlPressed && this.camera._useCtrlForPanning) {
    camera.inertialPanningX -= 1 / this.panningSensibility;
} else if (this._altPressed && this.useAltToZoom) {
    camera.inertialRadiusOffset += 1 / this.zoomingSensibility;
} else {
    camera.inertialAlphaOffset -= this.angularSpeed;
}
// × 4 directions (left, right, up, down)
// Want ctrl=rotate? → Rewrite the entire input class
✅ New — Reorder entries
// Filter out keyboard entries, add new ones:
camera.movement.inputMap = [
  ...camera.movement.inputMap
    .filter(e => e.source !== "keyboard"),

  // Ctrl+arrows → rotate (more specific = first)
  { source: "keyboard",
    modifiers: { ctrl: true },
    interaction: "rotate" },

  // Default arrows → pan
  { source: "keyboard",
    interaction: "pan" },
];

Before vs. After

Adjusting sensitivity / speed

❌ Legacy — Scattered, inconsistent properties
// Pointer input has its own sensitivity props:
pointerInput.angularSensibilityX = 1000;  // Higher = slower (!)
pointerInput.angularSensibilityY = 1000;
pointerInput.panningSensibility  = 1000;  // 0 = disable panning

// Keyboard input has different props:
keyboardInput.angularSpeed       = 0.01;  // Higher = faster
keyboardInput.panningSensibility = 50;    // Different default!

// Wheel has yet another pattern:
wheelInput.wheelPrecision        = 3;
wheelInput.wheelDeltaPercentage  = 0;    // 0 = use precision mode

// "Sensibility" = divisor (higher = slower)
// "Speed" = multiplier (higher = faster)
// 🤯 Inconsistent mental model!
✅ New — Unified, consistent model
// Global speed multiplier:
camera.movement.speed = 3.5;

// Per-axis speed (all multipliers, higher = faster):
camera.movement.panSpeed      = 1.0;
camera.movement.rotationXSpeed = 1.0;
camera.movement.rotationYSpeed = 1.0;
camera.movement.zoomSpeed     = 1.0;

// Per-entry sensitivity for fine control:
const entry = camera.movement
  .getEntry("pointer", "rotate");
if (entry) entry.sensitivity = 0.005;

// Or set per input map entry:
{ source: "pointer", button: 0,
  interaction: "rotate",
  sensitivity: 0.001 }  // ← per-binding

Before vs. After

Adding shift+drag to pan (new modifier behavior)

❌ Legacy — Subclass & rewrite
// No shift support in the input at all.
// You'd need to:

// 1. Subclass ArcRotateCameraPointersInput
class MyPointersInput extends
    ArcRotateCameraPointersInput {

  public override onTouch(point, offX, offY) {
    // 2. Duplicate the entire onTouch method
    // 3. Add your shift check
    if (this._shiftPressed) {
      camera.inertialPanningX += -offX /
        this.panningSensibility;
      // ...
    }
    // 4. Call super or duplicate rest of logic
  }
}
// 5. Remove old input, add new one
camera.inputs.remove(camera.inputs.attached.pointers);
camera.inputs.add(new MyPointersInput());
✅ New — One line
// addEntry auto-sorts by specificity!
camera.movement.addEntry({
  source: "pointer",
  button: 0,
  modifiers: { shift: true },
  interaction: "pan",
  sensitivity: 0.002
});

// Specificity score = 3 (button + modifiers + ...)
// Auto-inserted BEFORE the plain button:0 entry
// (score=1), so shift+left-click → pan,
// plain left-click → rotate (default).

// That's it. No subclassing.
// No rewriting input logic.
// No removing/re-adding inputs.

Before vs. After

Inertia: per-frame multiplication vs. decay-based velocity

❌ Legacy — Framerate-dependent
// In ArcRotateCamera._checkInputs():
this.inertialAlphaOffset *= this.inertia;
this.inertialBetaOffset  *= this.inertia;
this.inertialRadiusOffset *= this.inertia;

// inertia = 0.9 means "keep 90% each frame"
// At 60fps: after 1s → 0.9^60 = 0.18% remaining
// At 30fps: after 1s → 0.9^30 = 4.2% remaining
//
// 🐛 Same inertia value = different feel
//    at different frame rates!
✅ New — Framerate-independent
// In CameraMovement._calculateCurrentVelocity():
velocity *= Math.pow(
  inertia,
  prevFrameTime / 16.67  // normalize to 60fps
);

// At 60fps: deltaTime=16.67ms
//   → pow(0.9, 16.67/16.67) = 0.9
// At 30fps: deltaTime=33.33ms
//   → pow(0.9, 33.33/16.67) = 0.81
//
// ✅ Same visual deceleration regardless
//    of frame rate!

Future: Decay / Half-Life Model

@reedsyllas — forum.babylonjs.com/t/61001

"A half-life of 250ms means my camera will move half as fast after 250ms. What does inertia of 95% mean? I'd have to know the frame rate."

// The current velocity model opens the door to decay/half-life semantics:
const LOG_2 = Math.log(2);

// Convert half-life (e.g. 250ms) to decay constant
function decayFromHalfLife(halfLifeMs: number): number {
  return LOG_2 / halfLifeMs;
}

// Apply exponential decay — equivalent to current pow() but more intuitive to configure
function expDecay(current: number, target: number, decay: number, dt: number): number {
  return target + (current - target) * Math.exp(-decay * dt);
}

// Usage — "camera glides to a stop, halving speed every 250ms"
camera.movement.rotationHalfLife = 250; // milliseconds — intuitive!

Before vs. After

Custom rotation behavior (e.g., orbit around picked point)

❌ Legacy — Deep override
// Must subclass + rewrite _checkInputs or
// the entire pointer input class.
// Rotation logic is interleaved with:
//   - inertia decay
//   - panning logic
//   - zoom logic
//   - limit clamping
//
// Changing rotation means touching all of it.
✅ New — Replace one handler
// Replace just the rotate handler:
camera.movement.handlers.rotate = (dx, dy) => {
  myCustomOrbitLogic(camera, dx, dy);
};

// Or wrap the default to add logging:
const defaultRotate =
  camera.movement.handlers.rotate;
camera.movement.handlers.rotate = (dx, dy) => {
  console.log("rotate:", dx, dy);
  defaultRotate(dx, dy);
};

Handlers are source-agnostic

The rotate handler doesn't know if rotation came from a mouse drag, keyboard arrow, or touch gesture. The input mapping layer handles that distinction. This means your custom handler works for all input sources automatically.

How It All Connects

Data flow from physical input to camera state

cameraInteractions.ts ← Type definitions ↓ (types) CameraMovement (base class) ← cameraMovement.ts ├── inputMap[] + resolveInteraction() ├── Pixel accumulators (pan, rotation, zoom) ├── Velocity tracking + decay-based inertia ├── Speed config (per-axis + global) └── computeCurrentFrameDeltas() ↑ extends ├─ ArcRotateCameraMovement ← arcRotateCameraMovement.ts │ ├── handlers: { pan, rotate, zoom } (all plain functions) │ ├── Default inputMap for ArcRotate │ └── Used by: ArcRotateCamera │ └─ GeospatialCameraMovement ← geospatialCameraMovement.ts ├── handlers.pan: { start, update, stop } (lifecycle object) ├── Raycasting drag, lat-based damping └── Used by: GeospatialCamera Input Classes (existing, modified) ├── KeyboardMoveInput → resolveInteraction("keyboard", conditions) ├── PointersInput → resolveInteraction("pointer", conditions) └── MouseWheelInput → handlers.zoom(delta)

Performance: zero allocations in render loop

  • Input conditions objects are cached and mutated, not allocated per frame
  • Pixel accumulators use pre-allocated Vector3 instances
  • resolveInteraction() scan is ~6–8 string comparisons — negligible
